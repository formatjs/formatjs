# Instructions for Agents

Scope: this file applies to the entire FormatJS repository.

## Start With The Knowledge Base

Use `knowledge-base/` for architecture details before making non-trivial
changes. The highest-signal entry points are:

- `knowledge-base/001-repo-layout.md` for repository structure, workspace import
  rules, linting, hooks, workflows, and common commands.
- `knowledge-base/001a-bazel-toolchain.md` for the Bazel, TypeScript, Gazelle,
  generated-data, and packaging model.
- `knowledge-base/002-ts-package-dependency-hierarchy.md` and
  `knowledge-base/003-rust-crate-dependency-hierarchy.md` for dependency
  direction.
- `knowledge-base/007-package-intl-polyfills.md` plus `007a` through `007l` for
  polyfill data pipelines.
- `knowledge-base/010-script-conventions.md` for TypeScript generator and CLI
  script shape.
- `knowledge-base/011-generated-packages.md` for `@formatjs_generated/*`.

Treat files under `knowledge-base/migrations/` as historical plans unless the
live BUILD files confirm that the plan is still current.

## Hard Rules

- Never run `bazel clean` or `bazel clean --expunge` unless the user explicitly
  asks for it.
- Use Bazel for builds, tests, generated files, package assembly, and release
  binaries. Do not replace Bazel paths with ad hoc `npm`, `yarn`, `pnpm`,
  Cargo, or shell build paths.
- Run focused targets first, then broaden validation when the change affects
  shared macros, generated data, package boundaries, or public behavior.
- Run `bazel run //:gazelle` after TypeScript import graph or BUILD ownership
  changes.
- Use `bazel run -c opt` for Rust benchmarks; debug-mode timings are not useful.
- Keep generated TypeScript data in Bazel outputs through
  `generate_package_file()` / `formatjs_generated_package()` unless the repo
  already uses a checked-in Rust generated file path.
- Do not guess transitioned Bazel output paths. Use
  `bazel cquery --output=files <target>` when an exact artifact path matters.

## Common Commands

```bash
pnpm install
bazel build //...
bazel test //...
bazel test //packages/intl-localematcher:unit_test --test_output=all
bazel run //:gazelle
bazel run //docs:serve
bazel query 'kind(test, //packages/...)'
pnpm format
```

Root package scripts mostly delegate to Bazel. `pnpm install` is for dependency
installation; build and test work should still go through Bazel targets.

## Repository Shape

- `packages/` contains npm packages: polyfills, parsers, framework integrations,
  CLIs, and developer tooling.
- `crates/` contains Rust crates for the CLI and parser mirrors.
- `tools/` contains Bazel macros and TypeScript build/generation helpers.
- `docs/` is the Vike/React/MDX documentation site.
- `packages/generated/` and package-local generated-package targets feed
  `@formatjs_generated/*` data packages.
- `conformance-tests/` and package integration tests compare behavior against
  standards, ICU4J, JavaScript implementations, and Rust implementations.

## Bazel And TypeScript

The repo uses Bazel with bzlmod. `formatjs_library()` and `formatjs_test()` are
the local wrappers for TypeScript libraries and tests; Gazelle emits abstract
`ts_library` / `ts_test` rules and root `BUILD.bazel` maps them to those
wrappers.

Published packages use this rough pipeline:

```text
sources -> typecheck -> Rolldown bundle + d.ts bundle -> validation tests -> npm package
```

Internal TypeScript packages use typechecking plus per-file OXC transpilation.
Keep typechecking, transpilation, bundling, and package assembly as separate
Bazel concerns.

`tools/tsconfig.bzl` is the source of truth for TypeScript configs. Root
`tsconfig.json` is generated from it. Do not hand-tune package tsconfigs without
checking the generator path.

## Imports And Package Boundaries

- Root `package.json` defines `#packages/*` subpath imports.
- Internal package imports should be deep imports with runtime `.js`
  extensions, for example:

  ```ts
  import {GetOption} from '#packages/ecma402-abstract/GetOption.js'
  ```

- Do not add barrel `index.ts` exports for internal packages such as
  `ecma402-abstract`. Barrel exports are for public-facing packages.
- `ecma402-abstract` is internal and is not published to npm. It is consumed via
  `#packages/ecma402-abstract/...` and split across composite Bazel
  subpackages.
- Respect dependency layering from `knowledge-base/002-*`: lower layers should
  not depend on higher-level formatters, integrations, or CLI packages.

## Generated Data Packages

`@formatjs_generated/*` packages contain CLDR, IANA timezone, Unicode, collation,
and supported-locale data generated by Bazel. They are linked into
`node_modules` through `link_all_generated_packages()`.

When adding generated data:

1. Put generator scripts in the owning `packages/*/scripts/` Bazel package.
2. Use `generate_package_file()` for generated `.ts` outputs.
3. Add files to the relevant `formatjs_generated_package()`.
4. Regenerate the registry with `./scripts/generate_generated_packages.sh` when
   package targets are added or removed.
5. Import from `@formatjs_generated/<package>/<file>.js`.
6. Add `//:node_modules/@formatjs_generated/<package>` to Bazel deps.
7. Run `bazel run //:gazelle`.

Generated packages are bundled inline into published packages; they should not
be externalized by Rolldown.

## Polyfills And ECMA-402

Polyfills should follow ECMA-402 and the relevant LDML/CLDR behavior. Check the
specific `knowledge-base/007*.md` file before changing a polyfill data pipeline.

Shared expectations:

- Exports normally include `index`, `polyfill`, `polyfill-force`, and
  `should-polyfill`.
- Locale data must stay tree-shakeable when the package uses per-locale dynamic
  loading.
- Dynamic per-locale packages include numberformat, datetimeformat,
  pluralrules, displaynames, listformat, and relativetimeformat.
- Static-data packages include durationformat, segmenter, locale,
  getcanonicallocales, supportedvaluesof, and collator.
- Most polyfills depend on `ecma402-abstract` and `intl-localematcher`;
  numberformat, datetimeformat, and pluralrules also use `bigdecimal`.
- Validate with focused unit tests plus conformance or generated-data tests when
  behavior or data changes.

## Rust And CLI

The Rust crates mirror key TypeScript functionality:

- `formatjs_icu_skeleton_parser` mirrors `@formatjs/icu-skeleton-parser`.
- `formatjs_icu_messageformat_parser` mirrors
  `@formatjs/icu-messageformat-parser` and also has a WASM path.
- `formatjs_cli` is the Rust CLI implementation for extract, compile,
  compile-folder, and verify.

Use Bazel targets for Rust validation and release binaries. Useful focused
targets include:

```bash
bazel test //crates/formatjs_cli:formatjs_cli_test
bazel test //packages/cli/integration-tests:conformance_test
bazel run -c opt //crates/icu_messageformat_parser:parser_bench -- --bench
```

Release Please owns version/changelog PRs and GitHub release creation for npm
packages and Rust crates. Rust crate releases publish through
`.github/workflows/crates-release.yml` with crates.io trusted publishing. For
`formatjs_cli` binary artifacts, stay on the Bazel release path in
`.github/workflows/rust-cli-release.yml` and
`crates/formatjs_cli/release_binary.bzl`; that workflow uploads binaries and
checksums to the Release Please-created `formatjs_cli_v<version>` release
without replacing changelog notes. Keep `crates/formatjs_cli/Cargo.toml`,
`Cargo.lock`, and Bazel version metadata in sync, and verify the built binary's
`--version`.

## Scripts And Generators

For TypeScript scripts in `tools/` or `packages/*/scripts/`:

- Keep build-time scripts in their own Bazel package.
- Define a typed `Args` interface extending `minimist.ParsedArgs`.
- Put logic in `main(args: Args)`.
- Use an `import.meta.filename === process.argv[1]` entrypoint guard.
- Normalize repeatable flags with `[].concat(args.flag || [])`.
- Use real parsers for structured formats such as XML. Do not parse XML with
  tag or attribute regexes.

## Formatting, Hooks, And Commits

Formatting and linting are enforced by lefthook, oxfmt, oxlint, ast-grep,
buildifier, rustfmt, syncpack, and commitlint. `pnpm format` runs the pre-commit
hook stack for all files.

Commit messages use Conventional Commits:

```text
<type>(<scope>): <subject>
```

Use the full package name from `package.json` for scopes, a Rust crate name for
crate changes, or `deps` for repo-wide dependency/internal maintenance.
`@formatjs/ecma402-abstract` is not a valid commit scope because it is not an
npm package.

## Documentation

When a shipped API, workflow, generated-data model, or CLI behavior changes,
update both public docs and the relevant `knowledge-base/` page. For docs-page
additions, remember that `docs:sitemap_test` may require regenerating the
sitemap with `bazel run //docs:sitemap`.
